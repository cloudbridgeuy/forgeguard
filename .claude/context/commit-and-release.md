# Commit Conventions and Release Process

## Conventional Commits

All commits MUST follow [Conventional Commits](https://www.conventionalcommits.org/) for git-cliff changelog generation.

### Format

```
<type>(<scope>): <description>

[optional body]

[optional footer(s)]
```

### Types

| Type | Meaning |
|------|---------|
| `feat` | New feature or capability |
| `fix` | Bug fix |
| `refactor` | Code change that neither fixes a bug nor adds a feature |
| `docs` | Documentation only |
| `test` | Adding or updating tests |
| `ci` | CI/CD changes |
| `chore` | Maintenance (deps, tooling, config) |
| `perf` | Performance improvement |
| `style` | Formatting, whitespace (no logic change) |

### Scopes

Use the crate name suffix (not the full `forgegate_` prefix):

- `core`, `authn-core`, `authz-core`, `audit-core`
- `authn`, `authz`, `audit`
- `sdk`, `ffi-python`, `ffi-wasm`
- `control-plane`, `agent`, `cli`, `proxy`, `back-office`
- `xtask`, `ci`, `docker`

### Rules

- Scope is optional but strongly encouraged
- Description is imperative mood, lowercase, no period
- Breaking changes: add `!` after type/scope (e.g., `feat(sdk)!: change Guard API`)
- Multi-crate changes: use the most relevant scope, mention others in body

### Examples

```
feat(authn-core): add MagicLinkFlow typestate definitions
fix(authz): handle expired Cedar policy cache gracefully
refactor(sdk): extract Guard trait into separate module
docs(cli): add installation instructions
ci: add WASM build check to CI pipeline
chore(deps): bump aws-sdk-dynamodb to 1.x
```

## Release Process

Releases are managed via `cargo xtask release` with automatic version bump detection.

### Version Bump Logic

The xtask release command scans `git log` between the latest tag and HEAD, then determines the bump:

**Pre-1.0 (`0.x.y`):**

| Signal | Bump | Example |
|--------|------|---------|
| `!` in any commit | minor | `0.3.2 â†’ 0.4.0` |
| `feat` (no `!`) | patch | `0.3.2 â†’ 0.3.3` |
| Everything else | patch | `0.3.2 â†’ 0.3.3` |
| `--major` flag | major | `0.3.2 â†’ 1.0.0` |

**Post-1.0 (`x.y.z` where x >= 1):**

| Signal | Bump | Example |
|--------|------|---------|
| `!` in any commit | major | `1.3.2 â†’ 2.0.0` |
| `feat` (no `!`) | minor | `1.3.2 â†’ 1.4.0` |
| Everything else | patch | `1.3.2 â†’ 1.3.3` |
| `--major` flag | major | `1.3.2 â†’ 2.0.0` |

The regime is detected by reading the current version from the root `Cargo.toml`.

### Release Flow

1. `cargo xtask release` â€” scans commits, determines version bump
2. Updates all workspace `Cargo.toml` versions
3. Creates commit: `chore: bump version to X.Y.Z`
4. Creates and pushes `vX.Y.Z` tag
5. CI picks up the tag â†’ `release.yml` builds and publishes

### Changelog

Generated by `git-cliff` (configured in `cliff.toml`). Commits are categorized by type into sections (Features, Bug Fixes, Documentation, etc.). `chore(deps)` and `chore: bump version` commits are skipped.
